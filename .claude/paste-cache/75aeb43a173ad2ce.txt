## 1. Architecture cible — Verdent ↔ n8n ↔ Zapier

```
┌──────────────────────────────────────────────────────────────┐
│  VERDENT (MCP Client)                                        │
│  - Décide quoi faire, quand                                  │
│  - Appelle uniquement n8n via MCP HTTP                       │
└──────────────────────┬───────────────────────────────────────┘
                       │ MCP Streamable HTTP
                       │ https://agent.k3rnlabs.com/mcp-server/http
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  n8n (Orchestrateur + Gateway — couche durable)              │
│  - State machine, retries, idempotency keys                  │
│  - Audit log → Supabase                                      │
│  - Rate limiter Zapier (compteur tasks/jour)                 │
│  - Expose tools via MCP natif                                │
└──────────┬──────────────────────────┬────────────────────────┘
           │ Zapier MCP               │ Supabase REST / direct
           ▼                          ▼
┌──────────────────────┐    ┌────────────────────────┐
│  Zapier MCP Server   │    │  Supabase              │
│  • Gmail (send only) │    │  • Audit logs          │
│  • Slack (post only) │    │  • Task counter        │
│  allowlist: 4 tools  │    │  • Dossier data        │
│  no-delete enforced  │    └────────────────────────┘
└──────────────────────┘
```

**Qui orchestre quoi :**
- **Verdent** = décision (quel tool appeler, avec quels args)
- **n8n** = exécution durable (idempotency, retry ×3, audit, budget Zapier)
- **Zapier** = effecteur atomique final (Gmail/Slack), jamais appelé directement par Verdent

## 2. Contrat d'outils minimal

### Côté n8n → exposés à Verdent via MCP

| Tool | Input | Output | Passe par Zapier ? |
|---|---|---|---|
| `notify_slack` | `channel`, `message`, `level` (info/warn/alert) | `{ ts, ok }` | Oui — 1 task |
| `send_email` | `to`, `subject`, `body_md`, `reply_to?` | `{ messageId, ok }` | Oui — 1 task |
| `log_audit` | `dossierId`, `action`, `entity`, `metadata` | `{ id }` | Non — Supabase direct |
| `get_task_budget` | *(none)* | `{ used, remaining, reset_at }` | Non — compteur interne |
| `get_workflow_status` | `executionId` | `{ status, output? }` | Non — n8n API direct |

**Total : 5 tools exposés.** Pas de read Gmail/Slack (trop coûteux en tasks).

### Côté Zapier → appelé par n8n uniquement (jamais par Verdent)

| Tool Zapier | Action | Coût | Commentaire |
|---|---|---|---|
| `gmail_send_message` | Envoie un email | 1 task | Allowlisté |
| `gmail_create_draft` | Crée un brouillon | 1 task | Pas d'envoi auto |
| `slack_post_message` | Post dans un channel | 1 task | Allowlisté |
| `slack_send_dm` | DM à un user | 1 task | Allowlisté |

**Bloqués par défaut :** `gmail_delete`, `gmail_trash`, `slack_delete_message`, tout `*_update_*` sur Workspace.

## 3. Stratégie de sécurité

### Least privilege

```
Verdent  →  peut appeler : notify_slack, send_email, log_audit, get_*
n8n      →  peut appeler : 4 tools Zapier ci-dessus uniquement
Zapier   →  scoped OAuth : Gmail send + draft seulement, Slack post seulement
```

### Budget Zapier — circuit breaker (<100 tasks/jour)

n8n maintient un compteur `zapier_tasks_today` dans Supabase.

```
IF tasks_used >= 80  → WARNING au Slack internal + bloquer send_email
IF tasks_used >= 95  → HARD BLOCK toutes actions Zapier jusqu'à reset 00:00 UTC
```

Réserve : 80 tasks allouées aux workflows, 20 tasks de buffer.

### Secrets — 3 niveaux

```
DEV   : n8n env vars locales  — jamais commitées
STAGE : n8n credentials store — séparées de prod
PROD  : n8n credentials store — rotation trimestrielle
```

Variables à ne jamais mettre dans un workflow n8n :
- `ZAPIER_MCP_API_KEY`
- `GMAIL_OAUTH_TOKEN`
- `SLACK_BOT_TOKEN`

→ Toujours via **n8n Credentials** (chiffré au repos).

### Confirmations pour side-effects

Tout appel `notify_slack` ou `send_email` depuis Verdent doit inclure un `idempotency_key` (UUID généré côté Verdent). n8n rejette tout doublon sur 24h.

```typescript
// Pattern attendu dans le payload
{
  "idempotency_key": "uuid-v4",  // obligatoire
  "channel": "#kael-alerts",
  "message": "..."
}
```

### Étape 1 — Connecter Zapier MCP dans n8n *(~30 min)*
**Actions :**

2. Récupérer `ZAPIER_MCP_URL` + `ZAPIER_MCP_API_KEY`
3. Ajouter dans n8n : **Settings → Credentials → New → MCP Client** → pointer sur l'URL Zapier

**Done :** n8n peut lister les tools Zapier disponibles via un test MCP call manuel.

**Métriques :** latence connexion < 500ms, tools listés ≥ 4.

---

### Étape 2 — Créer les 5 workflows n8n stables *(~2h)*
Un workflow par tool exposé. Chaque workflow :
- Trigger : **MCP Tool trigger** (natif n8n v1.x)
- Validate input avec un **Code node** (Zod-like)
- Vérifie `idempotency_key` dans Supabase
- Exécute l'action (Zapier MCP node ou Supabase direct)
- Incrémente compteur tasks si Zapier utilisé
- Retourne output structuré

**Done :** `GET /mcp-server/http` liste les 5 tools. Test manuel de chaque via Verdent = succès.

**Métriques :** latence p50 < 2s (notify_slack), p50 < 3s (send_email), duplications = 0.

---

### Étape 3 — Hardening sécurité *(~1h)*
- Ajouter le circuit breaker tasks budget (workflow `check_zapier_budget` appelé avant chaque action Zapier)
- Configurer la Zapier allowlist (désactiver les tools non-listés dans le scope OAuth)
- Tester le hard block à 95 tasks

**Done :** Mock à 95 tasks → `notify_slack` retourne `{ ok: false, reason: "budget_exhausted" }`.

---

### Étape 4 — Wiring dans K3RN *(~1h)*
Dans `src/lib/n8n.ts`, ajouter :

```typescript
export async function notifySlack(channel: string, message: string, level: "info"|"warn"|"alert") {
  return callN8nTool("notify_slack", { 
    channel, message, level,
    idempotency_key: crypto.randomUUID()
  })
}

export async function sendEmail(to: string, subject: string, body_md: string) {
  return callN8nTool("send_email", {
    to, subject, body_md,
    idempotency_key: crypto.randomUUID()
  })
}

async function callN8nTool(tool: string, payload: object) {
  const res = await fetch(`${process.env.N8N_API_URL}/mcp-server/http`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${process.env.N8N_API_KEY}`
    },
    body: JSON.stringify({ tool, arguments: payload }),
    signal: AbortSignal.timeout(10000)
  })
  if (!res.ok) throw new Error(`n8n tool error: ${res.status}`)
  return res.json()
}
```

**Done :** `notifySlack("#kael-alerts", "Test K3RN", "info")` depuis le code → message reçu dans Slack.

## 5. Patch mcp.json + variables d'environnement

**`~/.verdent/mcp.json`** — état actuel déjà correct (Zapier n'est PAS dans Verdent) :

```json
{
  "mcpServers": {
    "supabase": { "...": "..." },
    "n8n": {
      "url": "https://agent.k3rnlabs.com/mcp-server/http",
      "headers": {
        "Authorization": "Bearer <JWT_MCP_TOKEN>"
      },
      "enabled": true
    }
  }
}
```

Zapier MCP n'est pas dans ce fichier — il est configuré **uniquement côté n8n** (Credentials). C'est intentionnel : Verdent ne connaît pas Zapier, il ne parle qu'à n8n.

**Variables `.env` à ajouter dans K3RN (Next.js runtime) :**

```bash
# n8n — déjà en place
N8N_API_URL=https://agent.k3rnlabs.com
N8N_API_KEY=<JWT_MCP_TOKEN>
N8N_WEBHOOK_SECRET=<secret_à_générer>

# Zapier — configuré dans n8n, pas ici
# (ne jamais mettre ZAPIER_* dans le .env Next.js)
```

**Variables à configurer dans n8n (Settings → Environment Variables) :**

```bash
ZAPIER_MCP_URL=https://mcp.zapier.com/api/mcp/s/<your_id>/mcp
ZAPIER_MCP_API_KEY=<zapier_mcp_api_key>
SUPABASE_URL=https://grdghxkowakgiqhybnrp.supabase.co
SUPABASE_SERVICE_KEY=<service_role_key>
ZAPIER_DAILY_TASK_LIMIT=80
```
