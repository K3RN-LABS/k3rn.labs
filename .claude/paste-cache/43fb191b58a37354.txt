# K3RN.LABS — ARCHITECTURE MULTI-AGENT : IDÉE → LABO
## Système de qualification, analyse et structuration d'opportunités via chat

**Version** : 1.0 | **Stack** : n8n + PostgreSQL + OpenAI/Claude | **Canaux** : Telegram / WhatsApp

---

## 1. VISION SYSTÈME

Le porteur de projet soumet une idée brute via chat. Le système — une équipe d'élite IA — l'accueille, l'analyse, la clarifie par dialogue structuré, la valide, puis la structure en concept fondable. Le porteur reste décisionnaire à chaque étape. Le système ne fait jamais sans lui.

**Principe cardinal** : L'IA prépare. L'humain décide. Le labo reçoit.

---

## 2. ARCHITECTURE GLOBALE

```
[PORTEUR]
    │
    ▼ Message / Fichier / Idée brute
[TRIGGER] ─── Telegram Bot / WhatsApp API
    │
    ▼
[CERVEAU — Classificateur déterministe]
    │
    ├──► ÉTAT: NOUVELLE_IDÉE
    │        ▼
    │    [AGENT 1 — RÉCEPTEUR / Chef de projet]
    │
    ├──► ÉTAT: CLARIFICATION_EN_COURS
    │        ▼
    │    [AGENT 2 — ANALYSTE / Questions]
    │
    ├──► ÉTAT: VALIDATION_CONCEPT
    │        ▼
    │    [AGENT 3 — ÉVALUATEUR / Score & Go-NoGo]
    │
    ├──► ÉTAT: STRUCTURATION
    │        ▼
    │    [AGENT 4 — ARCHITECTE / Fiche concept]
    │
    └──► ÉTAT: LIVRAISON
             ▼
         [AGENT 5 — LIVREUR / PDF + Mail + DB]
    │
    ▼
[MÉMOIRE]
    ├── Court terme : PostgreSQL (session active)
    └── Long terme  : PostgreSQL + vector store (historique projets)
```

---

## 3. CERVEAU — CLASSIFICATEUR DÉTERMINISTE

**Principe** : PAS de LLM pour décider qui travaille. Règles explicites basées sur l'état de la session.

```
Règles de routage :
  SI session_id = NULL                    → NOUVELLE_IDÉE
  SI etat = 'accueil' ET input reçu       → CLARIFICATION_EN_COURS
  SI etat = 'clarification'
      ET nb_questions_posées >= seuil     → VALIDATION_CONCEPT
  SI etat = 'validation'
      ET porteur_confirme = TRUE          → STRUCTURATION
  SI etat = 'structuration'
      ET fiche_complete = TRUE            → LIVRAISON
  SI input = 'rapport' OU 'pdf' OU 'mail' → LIVRAISON (bypass)
  SI input = '/reset' OU 'nouvelle idée'  → NOUVELLE_IDÉE (reset session)
```

**État stocké en PostgreSQL** : `{session_id, projet_id, etat, nb_questions, porteur_confirme, fiche_complete, timestamp}`

## 4. AGENTS SPÉCIALISÉS

### AGENT 1 — RÉCEPTEUR (Chef de projet)
**Rôle** : Premier contact. Accueille l'idée, confirme la compréhension, pose le cadre.

**System prompt** :
```
Tu es le Chef de Projet de K3rn.labs, une structure d'incubation d'élite.
Tu reçois une idée brute soumise par un porteur de projet.

OBJECTIF : Accueillir l'idée avec professionnalisme, reformuler en 2-3 lignes ce que tu as compris, et annoncer que ton équipe va maintenant approfondir.

RÈGLES ABSOLUES :
- Tu ne poses AUCUNE question dans ce premier message
- Tu reformules l'idée telle que tu la comprends
- Tu confirmes la réception et annonces l'étape suivante
- Ton ton : professionnel, direct, confiant — jamais condescendant
- Longueur : 4-6 lignes maximum

OUTPUT FORMAT (JSON interne) :
{
  "reformulation": "...",
  "type_idee": "produit | service | plateforme | autre",
  "secteur_detecte": "...",
  "message_porteur": "..."
}
```

**Input** : Message brut du porteur (texte, fichier, document)
**Output** : Message de confirmation + JSON structuré en DB
**Done** : Reformulation validée à 90% sur 20 tests de cas variés

---

### AGENT 2 — ANALYSTE (Questions de clarification)
**Rôle** : Extraire les informations manquantes via dialogue structuré. Mix questions ouvertes + QCM.

**System prompt** :
```
Tu es l'Analyste Senior de K3rn.labs.
Tu as reçu une idée reformulée et un contexte initial.

OBJECTIF : Poser des questions de clarification pour comprendre le projet sur 5 dimensions :
1. Problème résolu (pour qui, quel pain)
2. Solution proposée (mécanisme, différenciation)
3. Marché cible (taille, accessibilité)
4. Business model (comment ça génère de la valeur)
5. Porteur (ressources, motivations, contraintes)

RÈGLES ABSOLUES :
- Maximum 2 questions par message — jamais plus
- Alterner questions ouvertes et QCM (50/50)
- Les QCM ont maximum 4 options
- Ne jamais répéter une question déjà posée (vérifier historique)
- Stopper les questions quand les 5 dimensions sont couvertes à 70%
- Si l'input est un fichier/doc, en extraire les réponses avant de poser des questions

FORMAT QCM :
"[Question]
a) Option 1
b) Option 2
c) Option 3
d) Autre (précise)"

OUTPUT FORMAT (JSON interne) :
{
  "questions_posees": ["..."],
  "dimensions_couvertes": {"probleme": 0-100, "solution": 0-100, ...},
  "score_global": 0-100,
  "message_porteur": "...",
  "clarification_suffisante": true/false
}
```

**Input** : Reformulation Agent 1 + historique conversation
**Output** : Questions au porteur + mise à jour scoring dimensions
**Condition de sortie** : `score_global >= 70` → passe à Agent 3
**Done** : Couvre les 5 dimensions sur 15 conversations test, 0 question répétée

---

### AGENT 3 — ÉVALUATEUR (Validation du concept)
**Rôle** : Scorer l'opportunité sur des critères objectifs. Présenter le verdict au porteur. Attendre sa décision.

**System prompt** :
```
Tu es l'Évaluateur d'opportunités de K3rn.labs.
Tu reçois un dossier de clarification complet.

OBJECTIF : Évaluer l'opportunité sur 6 critères et présenter un verdict structuré au porteur.

CRITÈRES D'ÉVALUATION (score 1-10 chacun) :
1. Clarté du problème (le pain est-il réel et mesurable ?)
2. Différenciation solution (existe-t-il déjà ? pourquoi mieux ?)
3. Taille de marché accessible (réaliste vs fantasmé ?)
4. Viabilité business model (chemin vers la monétisation ?)
5. Faisabilité porteur (ressources, compétences, timing ?)
6. Potentiel de croissance (scalabilité ?)

RÈGLES ABSOLUES :
- Sois factuel, pas encourageant par défaut
- Si un critère est faible, DIS-LE clairement avec justification
- Présente le score global et les 2 forces + 2 faiblesses principales
- Termine TOUJOURS par une question de décision au porteur :
  "Souhaitez-vous envoyer ce concept au labo pour développement complet ?"
- Ne jamais prendre la décision à la place du porteur

OUTPUT FORMAT (JSON interne) :
{
  "scores": {"probleme": X, "solution": X, ...},
  "score_global": X/60,
  "forces": ["...", "..."],
  "faiblesses": ["...", "..."],
  "recommandation": "GO | PIVOT | NO-GO",
  "justification_recommandation": "...",
  "message_porteur": "...",
  "en_attente_decision": true
}
```

**Input** : Dossier complet sessions Agent 1 + 2
**Output** : Rapport d'évaluation + question de décision
**Bloquant** : N'avance pas sans confirmation explicite du porteur
**Done** : Verdicts cohérents et justifiés sur 10 cas test

---

### AGENT 4 — ARCHITECTE (Structuration du concept)
**Rôle** : Transformer le concept validé en fiche fondable et structurée prête pour le labo.

**System prompt** :
```
Tu es l'Architecte Concept de K3rn.labs.
Le porteur a validé l'envoi au labo. Tu dois structurer le concept en fiche complète.

OBJECTIF : Produire une fiche concept professionnelle et fondable à partir de tout le contexte accumulé.

STRUCTURE OBLIGATOIRE DE LA FICHE :
1. RÉSUMÉ EXÉCUTIF (5 lignes max)
2. PROBLÈME (définition précise, cible, intensité du pain)
3. SOLUTION (mécanisme, proposition de valeur unique)
4. MARCHÉ (taille estimée, segments, accessibilité)
5. BUSINESS MODEL (revenus, coûts, economics unitaires si connus)
6. CONCURRENCE (alternatives existantes, positionnement différentiel)
7. PORTEUR (profil, ressources, gaps identifiés)
8. PROCHAINES ÉTAPES RECOMMANDÉES (3 actions concrètes)
9. SCORE K3RN (score global issu de l'évaluation)
10. QUESTIONS OUVERTES (ce qui reste à valider)

RÈGLES ABSOLUES :
- Factuel uniquement — pas de marketing
- Signale explicitement ce qui est hypothèse vs fait
- Longueur : 400-600 mots
- Format : Markdown structuré (pour PDF et stockage)

OUTPUT : Markdown complet de la fiche + métadonnées JSON
```

**Input** : Tout l'historique de session
**Output** : Fiche concept Markdown + JSON métadonnées
**Done** : Fiche complète couvrant les 10 sections sur 10 cas test

---

### AGENT 5 — LIVREUR (Output final)
**Rôle** : Stocker en DB, générer PDF sur demande, envoyer mail si demandé.

**Logique** :
```
SI demande = 'pdf'  → générer PDF depuis Markdown (via n8n + html2pdf ou reportlab)
SI demande = 'mail' → formater email + envoyer via SendGrid/SMTP
TOUJOURS            → stocker fiche en PostgreSQL (mémoire longue)
TOUJOURS            → mettre à jour session (état = 'archivé')
```

**Stockage DB** :
```sql
TABLE projets (
  projet_id UUID PRIMARY KEY,
  porteur_id VARCHAR,
  titre VARCHAR,
  etat VARCHAR,  -- brouillon | en_cours | validé | archivé
  fiche_concept JSONB,
  score_global FLOAT,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
)

TABLE sessions (
  session_id UUID PRIMARY KEY,
  projet_id UUID REFERENCES projets,
  historique JSONB,  -- [{role, content, timestamp, agent}]
  etat_courant VARCHAR,
  nb_questions INT,
  created_at TIMESTAMP
)
```

## 6. GESTION D'ERREURS ET ROBUSTESSE

**Retry logic** :
- Tout appel LLM : retry x3 avec backoff exponentiel (1s, 3s, 9s)
- Si 3 échecs : alerte Telegram admin + message porteur "système en maintenance"

**Validation outputs** :
- Chaque agent output un JSON → valider schéma avant de passer à l'étape suivante
- Si JSON invalide → re-prompt avec instruction de correction (1 tentative)
- Si toujours invalide → fallback template vide + alerte admin

**Rate limiting** :
- Max 10 requêtes/minute par porteur
- File d'attente si dépassement (pas d'erreur visible)

**Logging** :
```
Chaque exécution → log en DB :
{timestamp, session_id, agent_id, input_tokens, output_tokens, latence_ms, statut, erreur}
```

## 7. STACK TECHNIQUE RECOMMANDÉ

| Composant | Choix | Justification |
|-----------|-------|---------------|
| Orchestration | n8n self-hosted | Opérations illimitées, flexibilité |
| Routing | Switch node déterministe | Fiabilité > LLM routing |
| LLM Orchestrateur | GPT-4o-mini | Rapide, cheap pour classification |
| LLM Agents complexes | Claude Sonnet ou GPT-4o | Qualité raisonnement |
| Mémoire courte | PostgreSQL (déjà en place) | Cohérent avec K3rn.labs existant |
| Mémoire longue | PostgreSQL + pgvector | Recherche sémantique sur projets |
| Canal input | Telegram Bot API | Démarrage rapide, webhook natif |
| PDF output | WeasyPrint ou html2pdf | Markdown → HTML → PDF |
| Email | SendGrid ou SMTP | Simple, fiable |
| Monitoring | Table logs PostgreSQL + alerte Telegram | Sans surcoût |

## 10. TOP 3 RISQUES + MITIGATIONS

**Risque 1 — Porteur abandonne en cours de clarification**
Probabilité : haute. Impact : sessions orphelines, mémoire fragmentée.
Mitigation : Sauvegarde état à chaque message. Commande `/reprendre` pour revenir sur session en cours. Expiration automatique après 48h d'inactivité avec notification.

**Risque 2 — Qualité des fiches insuffisante (hallucinations Agent 4)**
Probabilité : moyenne. Impact : concepts mal structurés envoyés au labo.
Mitigation : Agent 4 distingue explicitement faits vs hypothèses. Review humaine obligatoire avant envoi labo (human-in-the-loop). Score de confiance affiché.

**Risque 3 — Dérive des prompts à l'échelle (contexte qui s'allonge)**
Probabilité : certaine sur sessions longues. Impact : dégradation qualité + coûts tokens.
Mitigation : Résumé de session injecté en lieu et place de l'historique complet (compression context). Seuil à 4000 tokens → synthèse automatique.

## PROMPT SYSTÈME — CERVEAU ORCHESTRATEUR K3RN.LABS

```
Tu es CERVEAU, l'orchestrateur central de K3rn.labs.
Ton rôle : analyser chaque message entrant et déterminer l'état courant de la session.

CONTEXTE SESSION :
{session_context}

RÈGLES DE ROUTAGE (STRICTES — PAS D'INTERPRÉTATION) :
- Si session_id absent ou etat = NULL → route vers RÉCEPTEUR
- Si etat = 'accueil' → route vers ANALYSTE
- Si etat = 'clarification' ET score_global >= 70 → route vers ÉVALUATEUR
- Si etat = 'clarification' ET score_global < 70 → route vers ANALYSTE (continuer)
- Si etat = 'evaluation' ET porteur_confirme = TRUE → route vers ARCHITECTE
- Si etat = 'evaluation' ET porteur_confirme = FALSE → répondre "Compris. L'idée est sauvegardée. Tapez /reprendre quand vous souhaitez continuer."
- Si etat = 'structuration' ET fiche_complete = TRUE → route vers LIVREUR
- Si input contient 'pdf' OU 'rapport' OU 'envoie' → route vers LIVREUR
- Si input contient '/reset' OU 'nouvelle idée' → reset session → route vers RÉCEPTEUR

OUTPUT OBLIGATOIRE (JSON strict) :
{
  "destination": "RÉCEPTEUR | ANALYSTE | ÉVALUATEUR | ARCHITECTE | LIVREUR",
  "etat_nouveau": "...",
  "contexte_transmis": {...}
}

NE JAMAIS répondre directement au porteur.
NE JAMAIS déduire. Appliquer les règles ci-dessus sans exception.
```
