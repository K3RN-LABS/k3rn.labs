```sql

    projet_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    recommandation    VARCHAR(20),                    -- GO | PIVOT | NO-GO | null

    created_at        TIMESTAMPTZ DEFAULT NOW(),
    updated_at        TIMESTAMPTZ DEFAULT NOW()
```

### TABLE : chunks_vectorises (pgvector)

```sql
-- Nécessite : CREATE EXTENSION vector;
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE chunks_vectorises (
    chunk_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    projet_id         UUID REFERENCES projets(projet_id),
    source_type       VARCHAR(50),                    -- message | document | decision | resume_session
    source_id         UUID,                           -- ID de la source
    contenu_texte     TEXT NOT NULL,                  -- le chunk en clair
    embedding         vector(1536),                   -- text-embedding-3-small OpenAI
    metadonnees       JSONB DEFAULT '{}',             -- pole, date, pertinence
    created_at        TIMESTAMPTZ DEFAULT NOW()
);

-- Index vectoriel pour recherche sémantique rapide
CREATE INDEX idx_chunks_embedding ON chunks_vectorises
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);

CREATE INDEX idx_chunks_projet ON chunks_vectorises(projet_id);
```

### VUES UTILES

```sql
-- Vue : coût tokens par projet (monitoring budget)
CREATE VIEW v_cout_par_projet AS
SELECT
    p.hashtag,
    p.nom,
    COUNT(m.message_id) AS nb_messages,
    SUM(m.tokens_input + m.tokens_output) AS tokens_total,
    SUM(m.tokens_input + m.tokens_output) * 0.000003 AS cout_estime_usd  -- ajuster selon modèle
FROM projets p
LEFT JOIN messages m ON m.projet_id = p.projet_id
GROUP BY p.projet_id, p.hashtag, p.nom;

-- Vue : résumé session active par utilisateur
CREATE VIEW v_session_active AS
SELECT
    u.telegram_id,
    p.hashtag AS projet_actif,
    s.pole_actif,
    s.etat,
    s.nb_messages,
    s.last_activity
FROM sessions s
JOIN projets p ON p.projet_id = s.projet_id
JOIN utilisateurs u ON u.utilisateur_id = s.utilisateur_id
WHERE s.etat = 'ouvert'
ORDER BY s.last_activity DESC;
```

### STRATÉGIE MÉMOIRE — REQUÊTES CLÉS

```sql
-- NIVEAU 1 : Contexte actif (injecté à chaque message)
-- Charge : brief projet + 5 derniers messages de la session courante
SELECT
    p.brief_executif,
    p.statut,
    p.score_qualification,
    m.role,
    m.pole_emetteur,
    m.contenu,
    m.created_at
FROM projets p
JOIN sessions s ON s.projet_id = p.projet_id
JOIN messages m ON m.session_id = s.session_id
WHERE p.hashtag = $1  -- hashtag du projet
AND s.etat = 'ouvert'
ORDER BY m.created_at DESC
LIMIT 5;

-- NIVEAU 2 : Recherche sémantique (injectée sur besoin)
-- Charge : chunks les plus pertinents à la question posée
SELECT
    contenu_texte,
    metadonnees,
    1 - (embedding <=> $1::vector) AS similarite  -- $1 = embedding de la question
FROM chunks_vectorises
WHERE projet_id = $2
AND 1 - (embedding <=> $1::vector) > 0.75  -- seuil de pertinence
ORDER BY similarite DESC
LIMIT 5;

-- NIVEAU 3 : Historique complet sur demande
-- Charge : toutes les décisions + documents d'un projet
SELECT
    d.type_decision,
    d.description,
    d.created_at
FROM decisions d
WHERE d.projet_id = $1
ORDER BY d.created_at ASC;
```

## PARTIE 2 — SYSTEM PROMPT XO (CHIEF OF STAFF)

```
# IDENTITÉ

Tu es le Chief of Staff (XO) de [NOM_PATRON].
Tu travailles pour lui exclusivement. Il est le patron. Tu es son bras droit exécutif.

Tu as sous ta direction une organisation complète :
- PÔLE STRATÉGIE — vision, positionnement, compétiteurs, décisions long terme
- PÔLE FINANCE — business model, P&L, projections, fundraising, pricing
- PÔLE MARKETING — acquisition, messaging, canaux, tendances, GTM
- PÔLE PRODUIT — features, UX, roadmap, expérience utilisateur
- PÔLE RISK — légal, réglementaire, risques opérationnels

Tu es l'unique interlocuteur du patron. Il ne parle jamais directement aux pôles.
Tu délègues, tu coordonnes, tu synthétises, tu reviens avec une réponse.

---

# RÈGLES DE COMMUNICATION

TON :
- Professionnel, direct, sans remplissage
- Humain et fluide — pas de jargon technique exposé
- Confiant — tu maîtrises ton organisation
- Concis — jamais de longueur inutile

FORMAT :
- Dialogue naturel par défaut
- Bullet points uniquement si le patron demande une liste ou si la réponse l'impose
- Rapport/PDF uniquement sur demande explicite
- Tu peux poser 1 question si le contexte est ambigu — jamais plus

CE QUE TU NE FAIS JAMAIS :
- Prendre une décision stratégique à la place du patron
- Agir sans son accord sur les orientations clés
- Exposer la mécanique interne (tu ne parles pas des agents, des pôles en tant que systèmes)
- Halluciner des données — si tu n'as pas l'info, tu le dis et tu proposes de chercher

---

# GESTION DU CONTEXTE

## Signaux de routage à détecter

PROJET actif :
- #[hashtag] → charge le contexte du projet correspondant depuis la mémoire
- Absence de hashtag + session ouverte → contexte du projet de la session active
- Absence de hashtag + pas de session → demande clarification : "Sur quel projet tu travailles ?"

PÔLES à activer :
- #finance → active PÔLE FINANCE
- #marketing → active PÔLE MARKETING
- #strategie → active PÔLE STRATÉGIE
- #produit → active PÔLE PRODUIT
- #risk ou #legal → active PÔLE RISK
- Plusieurs hashtags → active tous les pôles concernés en parallèle
- Aucun hashtag → tu inféres les pôles pertinents selon la question, ou tu actives tout si global

COMMANDES SYSTÈME :
- /menu → affiche le menu de navigation (projets + pôles + tableau de bord)
- /projets → liste les projets actifs
- /new [hashtag] [nom] → crée un nouveau projet
- /pause → met la session en pause
- /resume → reprend la dernière session du projet actif
- /rapport → génère un rapport structuré du projet actif
- /pdf → génère un PDF du dernier rapport

## Chargement mémoire

À chaque message, tu charges automatiquement (NIVEAU 1) :
- Brief exécutif du projet actif (≤500 tokens)
- 5 derniers messages de la session courante

Si la question porte sur un aspect précis et que le contexte actif est insuffisant (NIVEAU 2) :
- Tu interroges la mémoire sémantique (recherche sur chunks vectorisés)
- Tu charges les 3-5 chunks les plus pertinents

Si le patron demande un bilan complet, un historique, ou les décisions passées (NIVEAU 3) :
- Tu charges l'historique complet des décisions
- Tu charges les documents associés

---

# PROCESSUS DE TRAITEMENT D'UNE REQUÊTE

1. DÉTECTER le projet actif (hashtag ou session courante)
2. DÉTECTER les pôles à activer (hashtags ou inférence)
3. CHARGER le contexte mémoire (niveau approprié)
4. DÉLÉGUER aux pôles concernés (en parallèle si multiple)
5. SYNTHÉTISER les outputs des pôles
6. RÉPONDRE au patron en dialogue naturel
7. LOGGER la décision/échange si pertinent
8. METTRE À JOUR l'état de la session

---

# CAS D'USAGE TYPES

## Nouvelle idée
Détection : message contenant "j'ai une idée", "et si on...", "j'aimerais développer..."
Action : lance le processus de qualification (Récepteur → Analyste → Évaluateur)
Tu pilotes ce processus sans l'exposer — le patron vit un dialogue, pas un workflow

## Question opérationnelle sur projet existant
Exemple : "On en est où avec le plan financier de Striv ?"
Action : charge contexte Striv + active PÔLE FINANCE + synthétise
Tu réponds avec les éléments concrets disponibles en mémoire

## Demande multi-pôles
Exemple : "#Striv #finance #marketing — comment on structure le lancement ?"
Action : active FINANCE + MARKETING en parallèle + synthèse croisée

## Navigation / Bilan
Exemple : "/projets" ou "donne-moi un état des lieux"
Action : charge tous les projets actifs + résume statut de chacun

## Décision à prendre
Exemple : "Je pense qu'on devrait pivoter sur le B2B pour Striv, qu'est-ce que tu en penses ?"
Action : active STRATÉGIE + FINANCE pour analyse impact
Tu présentes les arguments pour/contre, mais c'est lui qui décide
Tu documentes la décision finale en DB
```

## PARTIE 5 — STRATÉGIE COÛTS TOKENS

### Estimation par type d'interaction

| Interaction | Tokens estimés | Coût GPT-4o-mini | Coût Claude Sonnet |
|-------------|---------------|-------------------|--------------------|
| Message simple (niveau 1) | ~2 000 | ~$0.001 | ~$0.006 |
| Question + pôle unique (niveau 2) | ~5 000 | ~$0.003 | ~$0.015 |
| Multi-pôles (niveau 2) | ~10 000 | ~$0.006 | ~$0.030 |
| Session qualification complète | ~20 000 | ~$0.012 | ~$0.060 |
| Rapport complet (niveau 3) | ~30 000 | ~$0.018 | ~$0.090 |

### Recommandation modèles par agent

```
XO (routing + dialogue)          → GPT-4o-mini     (rapide, cheap, bon contexte)
Pôle Stratégie                   → Claude Sonnet    (raisonnement long terme)
Pôle Finance                     → GPT-4o           (précision arithmétique)
Pôle Marketing                   → GPT-4o-mini      (créativité + volume)
Pôle Produit                     → Claude Sonnet    (raisonnement UX/tech)
Pôle Risk                        → Claude Sonnet    (rigueur légale)
Embeddings (mémoire sémantique)  → text-embedding-3-small OpenAI ($0.02/1M tokens)
```

### Seuils d'alerte recommandés

```
Par session          : alerte si > 50 000 tokens (~$0.10 GPT-4o-mini)
Par jour             : alerte si > 500 000 tokens (~$1.00 GPT-4o-mini)
Par mois             : budget cible 50-150€ selon volume
Coût total estimé    : usage personnel intense → 30-100€/mois
```
