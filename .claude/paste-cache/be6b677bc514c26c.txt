Prochaine étape optimale
Blueprint technique — Project Context Analyzer v1

Objectif :
Transformer WindBoost d’un générateur générique en générateur contextuel.

1. Périmètre v1 (strict, non spéculatif)

Entrées supportées :

URL GitHub public

Branche cible

(Optionnel) Token GitHub

Sortie :

{
  "framework": "...",
  "language": "...",
  "runtime": "...",
  "package_manager": "...",
  "architecture_style": "...",
  "state_management": "...",
  "orm": "...",
  "database": "...",
  "test_framework": "...",
  "build_tool": "...",
  "monorepo": false,
  "notes": "Résumé synthétique"
}


Aucune génération LLM dans v1.
Uniquement détection déterministe + règles.

2. Architecture technique
2.1 Nouveau module
/core/project-context/
    ├── github-client.ts
    ├── repo-fetcher.ts
    ├── detectors/
    │     ├── framework.detector.ts
    │     ├── language.detector.ts
    │     ├── architecture.detector.ts
    │     ├── tooling.detector.ts
    │     └── testing.detector.ts
    ├── summarizer.ts
    └── types.ts


Chaque détecteur = responsabilité unique.

Atomicité stricte (aligné avec ta règle métier).

3. Pipeline d’analyse
Étape 1 — Fetch repo metadata

Utiliser GitHub REST API v3 :

GET /repos/{owner}/{repo}

GET /repos/{owner}/{repo}/contents/

GET /repos/{owner}/{repo}/contents/package.json

GET /repos/{owner}/{repo}/git/trees/{sha}?recursive=1

Documentation officielle :
https://docs.github.com/en/rest/repos/contents

Étape 2 — Détection déterministe
3.1 Détection framework

Règles exemple :

Signal détecté    Framework
next in dependencies    Next.js
react + vite    React + Vite
@angular/core    Angular
vue    Vue
fastify    Fastify
express    Express
nestjs    NestJS
3.2 Détection architecture

Heuristiques structurelles :

src/domain → DDD probable
src/app + src/infrastructure → Clean Architecture
pages/ ou app/ → Next.js routing
modules/ → Modular monolith


Détection basée sur structure tree.

3.3 Détection base de données

package.json :

prisma → Prisma

mongoose → MongoDB

typeorm → SQL

sequelize → SQL

drizzle-orm → Drizzle

3.4 Détection state management

redux

zustand

jotai

recoil

mobx

3.5 Détection tests

jest

vitest

playwright

cypress

testing-library

4. Résumé synthétique automatique

Pas de LLM.

Simple générateur template :

Le projet utilise {framework} avec {language}.
L’architecture semble être {architecture_style}.
La base de données est gérée via {orm}.
Le système de test repose sur {test_framework}.
Structure détectée : {pattern}.

7. Performance & contraintes

Limiter taille repo analysé (max 5MB tree metadata)

Timeout 10s

Cache résultat 24h

Rate limit GitHub : 60 req/h sans token, 5000 avec token

8. Impact produit

Après implémentation :

Avant

WindBoost = générateur générique

Après

WindBoost = générateur contextuel aligné au repo réel

Effet direct :

Tickets plus précis

Skills mieux recommandés

Moins d’erreurs de stack

Plus de crédibilité technique

9. Ce que tu ne dois PAS faire

Ne pas intégrer un LLM dès v1

Ne pas analyser tout le repo ligne par ligne

Ne pas générer des conclusions probabilistes

Ne pas complexifier avec AST parsing

v1 = détection robuste, simple, déterministe.
