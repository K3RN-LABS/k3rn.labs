Diagnostic

L’étape d’initialisation “KAEL → questions + réponses proposées” est visuellement réussie, mais stratégiquement fragile et logiquement incohérente à plusieurs endroits.

Les problèmes majeurs observables sur tes captures :

Collision de paradigmes UI

Tu as un chat linéaire (historique) + un modal de questions qui “prend le contrôle” en surcouche.

Résultat : l’utilisateur ne sait plus quel élément est “le vrai flux” (chat vs questionnaire). Le blur aide, mais ne résout pas la confusion de modèle mental.

Questions trop fermées, trop tôt

Première interaction : “Quel problème concret…” avec 4 options génériques (productivité, accès, coût, opportunité).

Ça force une catégorisation prématurée alors que l’utilisateur n’a pas encore posé son contexte. On perd le signal, et on biaise l’échange.

Incohérence de “prêt à démarrer”

Tu as une question encore affichée (“mesure principale de succès…”) en même temps que “Votre projet est prêt à démarrer”.

Ça casse la crédibilité de l’orchestrateur : l’utilisateur comprend que le système ne sait pas lui-même si c’est fini.

Risque de dérive sémantique

Les questions suivent un chemin qui peut sembler “auto-cohérent”, mais pas forcément aligné avec la réalité du projet. Si le texte initial (pitch) est long et interprété sans garde-fou, tu peux entrer dans un tunnel (“IA prédictive”, etc.) avant d’avoir validé le vrai besoin.

Faits / Hypothèses / Inférences / Recommandations en bref
Faits (depuis les captures)

UI : chat en arrière-plan + modal QCM au premier plan avec blur.

Plusieurs questions séquentielles ; réponses “1/2/3”.

Bouton “Entrer dans le workspace” + carte “Votre projet est prêt à démarrer” visible alors qu’une question subsiste.

Hypothèses (à valider)

Le système de questions suit un script (state machine) partiellement découplé du statut “ready”.

Les réponses proposées déclenchent un routing vers un lab (“LAB recommandé : …”) sans garde “completion”.

Inférences

Le produit mélange onboarding guidé et conversation ouverte sans hiérarchie explicite → friction cognitive.

Le “gating” (prêt/pas prêt) n’est pas une vérité unique → bugs d’état.

Recommandations

Clarifier le mode : soit “Assistant en conversation”, soit “Onboarding guidé”, mais pas un hybride flou.

Remplacer la première question fermée par une question ouverte + suggestions (chips) non contraignantes.

Introduire un state machine d’onboarding unique avec critères de complétion stricts : impossible d’afficher “prêt” tant que les champs requis ne sont pas remplis.

Angles morts

Absence de justification

Le système pose des questions mais n’explique jamais “pourquoi on te demande ça” ni “à quoi ça sert” (lab selection, cartes, routing). Sans explication, c’est perçu comme un quiz arbitraire.

Gestion du “skip”

Le bouton “Passer cette étape” existe : si l’utilisateur skip, quelle qualité minimale de données gardes-tu ? Sans stratégie, tu vas injecter du bruit dans la suite (labs/pôles).

Qualité des options

Les options sont trop génériques au début et deviennent très techniques (“IA prédictive”, “algorithmes de correction d’erreurs”) trop tôt. Ce gradient est inversé : tu veux du concret métier avant de parler de mécanismes.

Levier prioritaire (1 seul)

Unifier l’onboarding en une state machine stricte (source de vérité unique) + basculer le premier contact en “question ouverte assistée par suggestions”.

C’est le point qui corrige à la fois : confusion UI, qualité des inputs, cohérence “prêt à démarrer”, et robustesse future.

Plan d’exécution (étapes, dépendances, “done”, métriques)
Étape 1 — Définir l’Onboarding State Machine (backend + front)

But : une seule vérité : onboarding.status + onboarding.requiredFields.

Dépendances :

Schéma minimal côté DB (ou store serveur) : projectId, stepId, answers, status.

Done :

“Votre projet est prêt” n’apparaît que si status === COMPLETE.

Une question active = status === IN_PROGRESS.

Métriques :

0 occurrence de “ready” + question active simultanée (assertion + tests e2e).

Étape 2 — Refactor UX : “Open question first”, suggestions ensuite

But : ne pas enfermer le user.

Implémentation UX :

Première question : champ libre (“Décris ton projet en 2 phrases + objectif + contrainte principale”).

Afficher des suggestions cliquables (chips) qui complètent, pas qui remplacent.

Le modal devient un mode dédié (plein écran ou panneau latéral), pas une surcouche ambiguë.

Done :

L’utilisateur peut répondre sans choisir une option.

Métriques :

Taux d’abandon onboarding ↓, proportion de réponses “Autre” ↓.

Étape 3 — Critères explicites de “maturité suffisante pour lab”

But : “prêt” doit être explicable et stable.

Exemple de critères minimaux (non-tech) :

Problème (1 phrase)

Utilisateur cible (1 phrase)

Résultat attendu / metric (1)

Contrainte principale (1)

Done :

Un composant UI “✅ requis complétés / ❌ manquants”.

Métriques :

Diminution des labs incohérents (feedback interne).

Étape 4 — Alignement contenu : bannir les “solutions” trop tôt

But : éviter les tunnels (“IA prédictive”) avant d’avoir cadré.

Règle :

Avant d’évoquer une solution, le système doit avoir confirmé : problème + cible + outcome.

Done :

Tests : aucune question “solution” si les 3 champs ne sont pas remplis.

Top 3 risques + mitigations

Risque : perte de “wow effect” du modal QCM

Mitigation : garder les chips/suggestions + micro-interactions, mais sans forcer la réponse.

Risque : complexité d’état (bugs)

Mitigation : state machine unique + invariants testés (unit + e2e) + logs d’événements.

Risque : onboarding trop long

Mitigation : 4 champs max, auto-résumé en continu, possibilité de “continuer plus tard”.

Questions minimales (≤3)

Aujourd’hui, “lab recommandé” est-il déclenché par un score (règles) ou par une réponse LLM ?

Les réponses onboarding sont-elles déjà persistées quelque part (DB) ou uniquement en state front ?
